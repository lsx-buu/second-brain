# Agent 启动上下文

## 你是谁在服务

你是 Codex 或其他 coding agent。

这个知识库的主要用途，是让你在新对话、新线程、新任务里更快理解用户，减少重复解释，并把新经验沉淀回长期记忆。

## 第一原则

所有知识都服务于下一次相似任务。

如果一段内容不能帮助下一个 agent 更好地判断、执行或协作，它就不应该占用长期知识层。

## 最小读取顺序

新任务开始时，按这个顺序读：

1. [[_系统/Coding Agent 使用知识库协议|Coding Agent 使用知识库协议]]
2. [[10_知识库/偏好/AI 协作偏好总览|AI 协作偏好总览]]
3. [[90_索引/问题到方法群路由|问题到方法群路由]]
4. [[90_索引/方法复用验证队列|方法复用验证队列]]
5. 当前任务命中的方法群或项目案例。

不要默认读取整个知识库。

## 用户协作偏好摘要

用户更喜欢 agent 像能落地的协作者，而不是只给建议的顾问。

默认行为：

- 先读现有材料，再判断。
- 能落地就直接推进。
- 任务明确时主动执行、验证、迭代。
- 知识库内容默认中文、无代码、偏方法论。
- 视觉、汇报、UI 类任务要关注正式感、真实感和业务可信度。
- 环境、接口、服务器问题要分层排查，不要只猜。
- 不要在工作对话里强行复盘；复盘应放到单独时刻，并保持轻量。
- 不要在工作对话里显式打断用户导入新材料；明显有价值的材料先放 [[00_收件箱/夜间复盘候选|夜间复盘候选]]，晚上再通过微信复盘处理。

## 常见任务路由

### 新项目或 AI 原型

先看 [[10_知识库/方法论/方法群/AI 原型验证方法群|AI 原型验证方法群]]。

判断输入、智能、呈现和运行是否形成闭环。

### 汇报、PPT、业务材料

先看 [[10_知识库/方法论/方法群/业务可信表达方法群|业务可信表达方法群]]。

判断是否有证据链、字段来源、真实素材和追问防线。

### 实验、模型、指标

先看 [[10_知识库/方法论/方法群/实验与模型审计方法群|实验与模型审计方法群]]。

判断评价口径、切分稳定性、可学习性和可观测性。

### 环境、服务器、接口

先看 [[10_知识库/方法论/方法群/工程运行方法群|工程运行方法群]]。

判断本地、网络、认证、服务、配置和反馈链路。

### 知识库、复盘、沉淀

先看 [[10_知识库/方法论/方法群/知识沉淀方法群|知识沉淀方法群]]。

如果是导入一段新材料、文档或聊天摘录，先看 [[_系统/新内容导入 SOP|新内容导入 SOP]]。

如果用户不想显式导入材料，先看 [[_系统/隐式材料收件箱与夜间微信复盘规则|隐式材料收件箱与夜间微信复盘规则]]。

如果当前任务发生在服务器或远程 agent，先看 [[_系统/GitHub 同步与服务端读取规则|GitHub 同步与服务端读取规则]]，确认知识库是否已同步到最新版本。

再看 [[_系统/Codex 对话沉淀 SOP|Codex 对话沉淀 SOP]]。

### 学习、备考、长期协作

先看 [[10_知识库/方法论/方法群/学习与协作方法群|学习与协作方法群]]。

如果用户的目标是让 AI 反过来训练判断力、暴露盲点或做复盘，先看 [[10_知识库/方法论/AI 反哺式学习与问题解决|AI 反哺式学习与问题解决]] 和 [[_系统/AI 反哺复盘 SOP|AI 反哺复盘 SOP]]。

再看 [[10_知识库/偏好/AI 协作偏好总览|AI 协作偏好总览]]。

## 任务结束时必须判断

完成任务后问：

- 这次是否产生了可复用判断？
- 是否命中 [[90_索引/方法复用验证队列|方法复用验证队列]]？
- 是否应该补旧卡，而不是新建卡？
- 是否暴露了新的用户偏好？
- 是否只需要留下一个复盘候选，而不是在工作线程里展开复盘？
- 是否需要写交接，方便下一个 agent 接手？

## 写回优先级

优先补旧卡。

其次更新索引、验证队列或偏好。

最后才新建方法卡。

如果只是一次性事实，放弃写入。
