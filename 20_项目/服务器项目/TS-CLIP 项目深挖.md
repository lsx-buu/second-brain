# TS-CLIP 项目深挖

## 项目坐标

- 远程主机：20。
- 项目路径：/root/data1/lisixing/TS-CLIP。
- 相关线程：优化 HCR MoE 门控和执行模块；协变量约束流匹配。
- 项目状态：长期研究型项目，包含多条模型、审计和论文主线。

## 一句话理解

TS-CLIP 当前最重要的价值，不只是“把地铁客流预测误差降下来”，而是把时序模型在真实地铁数据上为什么失败、哪些失败可修、哪些失败来自信息不可观测，逐步变成可解释、可审计、可写成论文的研究系统。

## 主要研究线一：OD 预测与可观测性缺口

当前 OD 预测任务围绕少量站点和四个方向 OD 展开，评估重点是 MAPE@100，并采用 feature-macro 的口径。

已经读到的关键结果：

- base_v2 only 约 15.45。
- base_v2 加 drop_horizon BRC 约 14.60。
- low-cover conservative repair 约 14.58，是目前读到的最好可信 clean 结果。
- dirty diagnostic oracle 约 8.85，说明错误理论上可以被压缩。
- JCH-OD v2 最好可信审计约 14.71，不能超过 BRC 主线。

由此形成的核心判断是：内部信息存在可观测性缺口。也就是说，当前历史客流、站点预测、检索先验和日历信息不足以解释全部 hard regime。继续在内部 clean 特征上堆 router、selector 或小模型，收益大概率不稳定。

下一步更适合把论文主线转成：真实 OD 预测中存在 external observability gap，需要航班、天气、运营调整、大型活动、调休强度等外部动态变量。

## 主要研究线二：HCR、MoE 与局部形态修正

HCR/MoE 这条线的起点是基于当前最优版本继续优化 gate 和后面的执行模块，尤其关注 gate 权重和执行模块是否联动。

最新审计把问题从“模型结构怎么改”推进到“错误到底长什么样”：

- 全局斜率审计发现，预测斜率与真实斜率相关性很高，模型通常知道大方向。
- 只修整条 42 点曲线的斜率，上界改善大约只有 1.2 到 1.35 MAPE。
- 因此全局 slope correction 太粗。
- 把未来 42 点拆成 6 个 7-step block 后，发现局部 block 的 level、slope、低频形状才是更大的错误来源。
- 可学习性审计进一步说明，当前可见特征最能学的是 block-local level，其次是 slope，curvature 暂时不稳。

所以当前更稳的研究角度不是完整低频形状头，而是更收敛的 block-local level-slope correction：先学局部水平偏移，再小幅加入局部斜率，暂时不要训练曲率。

## 主要研究线三：弱先验和可靠性

Chronos、Sundial、TSFM 这类时序大模型先验，在这些对话里反复出现一个共识：

它们不适合直接当强预测器，更适合当弱先验、方向教师、可靠性信号或残差修正提案器。

原因是，真实地铁客流有强非平稳、节假日、空间异质性、峰值突变和外部事件。预训练时序模型容易带着平滑、稳定、周期性过强的先验进入目标域，导致峰值被抹平、极端值偏低或局部状态切换错位。

更稳的用法是：让大模型只提出可能的修正方向和不确定性，由一个很小的 permission-budget 层决定是否允许移动、移动多少，并用峰值和长 horizon 做硬保护。

## 这个项目最有价值的高维沉淀

TS-CLIP 训练的是“研究失败机制”的能力。

它提醒我：当一个模型不够好时，不要马上换更大的模型。更好的顺序是：

1. 先固定评价口径。
2. 再找 hard regime。
3. 再定义代理信号。
4. 再做 oracle 上界审计。
5. 再做可学习性审计。
6. 最后才决定是否训练新模块。

如果 oracle 很强但可学习性不稳，那它是论文证据或诊断线索，不一定是可部署方法。

## 这个项目最值得继续沉淀的方向

不是继续记录模型名和实验线，而是继续提炼：

- 如何识别可观测性缺口。
- 如何区分 oracle 上界和可学习性。
- 如何判断一个干净结构能否升为主方法。
- 如何处理预训练大模型在强 domain shift 下的弱先验。
- 如何保护正式评价口径，不被局部改善误导。
