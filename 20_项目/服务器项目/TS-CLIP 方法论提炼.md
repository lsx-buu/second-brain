# TS-CLIP 方法论提炼

## 这个项目真正提供了什么

TS-CLIP 不应该在知识库里被保存成一堆模型名、指标和决策流水账。

它真正提供的是一套关于“真实世界预测为什么失败”的判断框架。

如果项目文件全部删除，最值得保留的是这几个方法：

- 区分模型能力不足和信息不可观测。
- 区分 oracle 上界和可学习信号。
- 区分验证集好看和测试集可靠。
- 区分弱先验、强预测器和风险信号。
- 区分局部点改善和正式评价口径改善。

## 方法一：先找失败机制，再训练模块

不要因为模型错了就直接加模块。

先问：

- 错误集中在哪些状态。
- 这些状态是否有共同外部原因。
- 修正这个错误的 oracle 上界有多大。
- 当前可见信息能不能学到这个修正。
- 修正后是否伤害正式指标。

这比直接堆模型更重要。

## 方法二：oracle 只能证明“值得研究”，不能证明“可以部署”

TS-CLIP 里很典型的一点是：某些局部形状或曲率的 oracle 很强，但可见特征并不一定能稳定学到。

这可以迁移到任何项目：

当你看到“如果知道某个隐藏答案就能大幅改善”时，不要立刻做模型。先问这个隐藏答案在真实决策时刻有没有可学习代理。

## 方法三：发现可观测性缺口时，不要硬包装模型创新

当 dirty oracle 明显好于 clean 方法，而多个 clean 方法都卡在相近水平，就要怀疑不是模型不够，而是输入里缺世界信息。

这时论文或产品主线应该转向：

当前系统缺什么外部可观测变量。

而不是继续说：

我们提出一个更复杂的模型。

## 方法四：弱先验不要强行当主模型

Chronos、Sundial、TSFM 这类大模型先验，在真实地铁场景里可能有方向或风险信息，但不能默认作为强预测分支。

更稳的处理是：

- 当方向信号。
- 当不确定性信号。
- 当风险过滤信号。
- 当小幅残差提案。

不要让它直接主导主预测。

## 方法五：评价口径是研究对象的一部分

row-average 改善、局部样本改善、某个切分改善，都不等于正式目标改善。

在任何预测项目里，评价口径都不是最后报告才写的东西，而是设计方法时就要保护的约束。

## 可迁移到哪些场景

- 地铁客流、交通、能源、销量等真实世界时序预测。
- 数据缺外部变量但模型越堆越复杂的项目。
- 有 oracle 诱惑的研究项目。
- 预训练模型迁移到强 domain shift 场景。
- validation 好看但 test 不稳定的论文项目。
