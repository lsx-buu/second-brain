# 远程环境与交付链路项目集合

## 项目集合说明

这组对话围绕“项目如何在不同机器之间运行和交付”展开。

它包括 SSH、VPN、EasyConnect、免密登录、服务器同步、GitHub 私有仓库、服务重启、配置保护等问题。

本轮新增了一个更具体的分支：服务器研究项目。

代表项目是 watchbot_online 和 TS-CLIP。它们不仅需要“能连上服务器”，还需要能接管研究现场、读懂最近实验、保护评价口径、避免重复跑已经被否定的路线。

## 共同目标

让项目不只在当前机器上能跑，还能被迁移、同步、部署、更新和恢复。

## 关键问题

- 当前流量到底走哪里？
- 本地 VPN 是否影响服务器？
- SSH 是网络问题还是认证问题？
- 服务器是否需要安装 Codex？
- 文件同步和 GitHub 交付怎么做？
- webhook、token、配置文件如何不丢？
- 修改后是否需要重启服务？

## 高维能力

这类项目训练的是“运行环境可解释性”。

你不只是要一个命令，而是要知道系统当前处在哪个层次：

- 本地机器。
- VPN。
- 远程服务器。
- Git 仓库。
- 运行服务。
- 模型接口。
- 配置文件。

## 工作原则

远程交付不能靠猜。

长期值得沉淀的不是某个路径，而是排障顺序：

- 先区分本地、VPN、远程服务器、代理、目标服务。
- 再判断是网络、认证、端口、权限、配置还是服务进程问题。
- 最后才记录具体解决动作。

路径、服务名、配置位置只有短期价值；分层排障方法才有迁移价值。

## 第一性原理校验

这张卡能指导下一次相似项目：当一个问题跨机器、跨网络、跨模型接口时，不要直接试命令，先画清流量和责任层。

如果问题只发生在纯本地单文件任务里，这张卡不适用。
