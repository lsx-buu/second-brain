# GitHub 同步与服务端读取规则

## 目标

用 GitHub 私有仓库作为知识库同步层，让本地 Codex、微信复盘服务和服务器上的 coding agent 能读取同一套第二大脑。

GitHub 在这里不是公开展示入口，而是跨设备、跨服务端的同步通道。

## 推荐形态

知识库应放在私有仓库。

仓库里保留：

- 方法论卡。
- 偏好卡。
- 项目方法论提炼。
- 系统规则。
- 复盘卡。
- Agent 启动上下文。

仓库里不应保留：

- 密码、token、密钥和凭证。
- 完整聊天记录。
- 本地工作区状态。
- 大型临时附件。
- 只对一台电脑有效的缓存文件。

## 本地到 GitHub

本地知识库变更后，先检查：

- 是否有敏感信息。
- 是否把一次性现场事实误写成长期知识。
- 是否有断链。
- 是否保持无代码、中文、偏方法论。

确认后再提交并推送到 GitHub。

## 服务端读取

服务器或远程 agent 应 clone 私有仓库，然后从仓库根目录读取 `AGENTS.md`。

不要依赖本机绝对路径。

服务端启动任务前，先更新：

- 拉取 GitHub 最新版本。
- 读取 `90_索引/Agent 启动上下文.md`。
- 按问题路由读取相关方法群。

## 服务端写回

服务端如果产生新经验，不要只留在服务端对话上下文。

写回顺序是：

- 先放 `00_收件箱/夜间复盘候选.md`。
- 复盘后补旧卡。
- 必要时新增复盘卡。
- 最后提交并推送。

## 冲突处理

如果本地和服务端都改了知识库，优先保留人工整理过的结构。

冲突文件不要机械合并。

先判断哪一段能服务下一次相似任务，再决定保留、合并或丢弃。
