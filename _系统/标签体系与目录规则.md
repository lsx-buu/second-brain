# 标签体系与目录规则

## 第一性原理

这些东西，对我下一次同样类型的、或者有一定相似性的项目，能不能产生参考价值和指导意义？

这是所有目录、标签、卡片取舍的最高标准。

如果一张卡离开当前项目就失效，它就不是长期知识。

对本知识库来说，“下一次”通常指下一次 Codex 或其他 coding agent 接手任务。

因此卡片要尽量让 agent 能调用，而不只是让人看懂。

## 为什么要重构标签

标签不是给每个名词贴一个纪念章。

标签的作用是召回、聚合和复盘。一个标签如果只出现一次，通常不适合作为标签，更适合作为正文里的关键词、文件名或双链。

当前知识库的问题是：标签太像临时关键词，导致看起来信息很多，但无法真正筛选。

## 稳定标签族

以后优先使用这几类标签：

- 系统：用于标记第二大脑自身、模板、规则、自动化。
- 类型：用于标记项目、方法论、复盘、索引、偏好、模式、自动化候选。
- 类型/任务剧本：用于标记 agent 可直接执行的开工剧本。
- 类型/反模式：用于标记反复导致返工的失败模式。
- 项目：用于真正需要长期追踪的项目，只有多张案例卡或方法论卡都会用到时才创建。
- 领域：用于区分地铁运维、地铁客流、策略研究、AI 互动、远程环境、内容生产等。
- 能力：用于标记可迁移能力，例如可信审计、项目接管、证据链、原型验证、环境排障、模型可靠性。
- 状态：用于聚合进行中、生效、待补全等状态。

## 不再使用的标签方式

- 不把一次性名词变成标签。
- 不把文件名重复写成标签。
- 不把每个具体功能都做成标签。
- 不把情绪化判断写成标签。
- 不同时使用多个意思相近的标签。

例如“真实素材”“视频素材”“筛选规则”更适合放在正文或双链里，稳定标签应该是“能力/证据链”或“领域/地铁运维”。

## 项目提炼规则

重要项目不能只放一张总览，也不应该优先变成运行手册。

真正值得沉淀的项目，至少要向上提炼出：

- 项目暴露的普遍问题。
- 项目里的关键误区。
- 能迁移到其他项目的判断原则。
- 不能迁移、应该丢掉的局部事实。
- 由项目抽象出的正式方法论卡。
- 下一次遇到类似问题时的判断清单。

如果一个项目节点离开原始文件就失效，它就不是好的知识节点。

项目文件、路径、运行状态只在短期接管时有用；第二大脑默认不把它们作为长期知识核心。

## 节点压缩规则

不要因为一个项目下面有很多功能、实验、素材或追问，就自动创建很多兄弟节点。

新增节点前先检查：

- 它和已有卡片是否在回答同一个问题。
- 它的结构是否只是换了对象名称。
- 它是否只是某个方法的案例。
- 它能否补进已有母卡，而不是单独成卡。

如果多张卡高度同质，优先合并成上位方法卡，把具体内容放进案例段落。

只有当新内容带来新的机制、边界、反模式或判断动作时，才新增独立节点。

## 方法论卡规则

方法论卡不能只是漂亮总结。

一张好方法论卡至少要回答：

- 适用场景是什么。
- 不适用场景是什么。
- 判断清单是什么。
- 失败信号是什么。
- 关联了哪些项目。
- 下一次遇到类似问题时怎么直接复用。

## 方法证据成熟度规则

每张核心方法卡都应该写明“证据成熟度”。

成熟度不是好坏评分，而是证据状态：

- 系统级成熟：由整个知识库运行和复盘证明。
- 跨项目成熟：被多个项目族或不同类型项目共同支撑。
- 领域内成熟：在一个领域或一类项目中证据较强。
- 单项目强证据：来自一个项目的证据很强，但不能过度泛化。
- 待复用验证：有迁移潜力，但还需要后续案例验证。

方法卡要升级成熟度，必须新增跨项目证据、跨领域证据、反例或真实复用后的修正。

同一个项目里的更多细节，通常不能让方法卡升级成熟度。

新增项目、文档或对话沉淀前，先查看 [[90_索引/方法复用验证队列|方法复用验证队列]]。

如果新材料能验证队列里的方法，优先补旧卡证据，不新建平行方法卡。

## 方法群规则

方法论目录不应该只是方法卡平铺。

每张方法卡都应该尽量归入一个方法群。方法群回答上位问题，方法卡提供具体判断动作，项目卡提供案例证据。

方法群不能只是链接列表。每个方法群至少要有：

- 上位问题。
- 适用场景。
- 内部层级。
- 核心方法。
- 代表案例或关联入口。

每张具体方法卡应该显式写出“所属方法群”，方便从方法回到母题。

方法群应该像问题地图，而不是目录。每个方法群还应尽量补充：

- 触发信号：什么时候该进入这个方法群。
- 阅读路线：先看哪张卡，再看哪张卡。
- 分叉判断：遇到混合问题时转向哪个方法群。
- 不该放进这里：防止方法群变成杂物箱。

当前方法群包括：

- 知识沉淀方法群。
- AI 原型验证方法群。
- 业务可信表达方法群。
- 实验与模型审计方法群。
- 工程运行方法群。
- 学习与协作方法群。

只有当新方法无法归入现有方法群，并且暴露了新的上位问题时，才新增方法群。

## 任务剧本规则

任务剧本放在 `10_知识库/任务剧本`。

任务剧本不是新方法，而是方法的执行版。

当任务已经明确、agent 需要直接开工时，优先读任务剧本；当任务边界不清楚或需要理解原理时，再读方法群和方法卡。

一张任务剧本至少要包含：

- 什么时候读。
- 开工前先看什么。
- 第一轮动作。
- 不要做什么。
- 有效信号。
- 失败信号。
- 写回判断。

任务剧本不保存现场路径、脚本参数和一次性文件状态。

如果剧本越写越长，说明它在变成教程或运行手册，应该拆回方法卡、项目案例或系统 SOP。

## 反模式规则

反模式放在 `10_知识库/反模式`。

反模式不是为了记录失败，而是为了让 agent 识别“当前任务正在往错误方向走”。

一条反模式至少要包含：

- 表现。
- 用户可能怎么反馈。
- 应该切换到哪个任务剧本或方法。
- 不应该继续做什么。

如果用户连续两次指出同一类问题，agent 应先查反模式，再继续执行。

反模式不要拆得太碎。优先放进 [[10_知识库/反模式/反模式总览|反模式总览]]，只有当某个反模式跨多个项目反复出现，才单独成卡。

## 自动沉淀规则

自动沉淀时，优先补已有卡片，不要轻易新建浅卡。

Agent 使用规则见 [[_系统/Coding Agent 使用知识库协议|Coding Agent 使用知识库协议]]。

人工执行流程见 [[_系统/Codex 对话沉淀 SOP|Codex 对话沉淀 SOP]]。

新材料进入知识库前见 [[_系统/新内容导入 SOP|新内容导入 SOP]]。

成长型任务复盘见 [[_系统/AI 反哺复盘 SOP|AI 反哺复盘 SOP]]。

写入前先走 [[90_索引/问题到方法群路由|问题到方法群路由]]，判断内容属于哪个方法群，还是只适合进入收件箱。

新卡必须满足至少两项：

- 有明确来源。
- 有至少两个双链。
- 有可迁移判断。
- 使用稳定标签。
- 能抽象出可迁移方法。
- 即使原项目文件删除也仍然有价值。
