# 工程运行方法群

## 上位问题

项目如何在真实环境里跑起来、连得上、迁得走？

## 适用场景

- 远程服务器连接。
- VPN、SSH、端口、服务启动问题。
- 模型接口不稳定。
- 本地和服务器之间交付链路混乱。
- 原型从能跑变成能演示、能迁移。
- 本地知识库需要通过 GitHub 让服务端 agent 可见。

## 触发信号

- 本地能跑，服务器不能跑。
- 连接、认证、端口、代理、接口其中一层不清楚。
- AI 功能没有反馈，用户不知道请求是否发出。
- 换环境后又要重新排障。

## 内部层级

先分层定位连接和环境问题，再标准化接口配置，最后沉淀成可迁移的运行约束。

这一组方法只处理“真实环境里怎么稳定跑起来”，不承担项目方法论提炼。

## 阅读路线

1. 先看 [[10_知识库/方法论/网络与远程环境排障方法|网络与远程环境排障方法]]，把链路分成本地、网络、认证、服务、配置。
2. 如果问题和模型服务相关，看 [[10_知识库/方法论/模型接口配置标准化|模型接口配置标准化]]。
3. 如果问题和 SSH、GitHub、deploy key 或远端同步相关，仍先按网络排障方法分层确认。
4. 如果排障过程暴露出可复用原则，再回到 [[10_知识库/方法论/方法群/知识沉淀方法群|知识沉淀方法群]]。

## 不该放进这里

- 纯业务判断。
- 单次报错截图。
- 没有迁移价值的临时路径。
- 模型效果差但环境已经正常的问题。
