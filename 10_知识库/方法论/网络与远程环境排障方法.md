# 网络与远程环境排障方法

## 解决什么问题

远程服务器、VPN、EasyConnect、SSH、反向代理和 Codex 远程环境经常会交织在一起，表面错误可能相同，真实原因却在不同层。

## 核心结论

排障时要分层确认，不要直接猜。

先确认网络路径，再确认端口，再确认认证，再确认服务，再确认 Codex 或模型请求链路。

## 常见场景

- SSH 连接超时。
- 多个 VPN 同时开启。
- EasyConnect 链接打不开。
- 远程端口转发失败。
- 服务器访问模型服务返回 403。
- 本地可用但服务器不可用。
- 虚拟网卡影响路由。
- GitHub deploy key、SSH alias 和远端同步权限混在一起。

## 排障顺序

先问清楚：

- 本地能不能访问目标。
- 服务器能不能访问目标。
- 流量到底走本机、VPN、远程服务器还是反向代理。
- 是连接失败、超时、认证失败，还是服务拒绝。

再逐层确认：

- 路由。
- VPN 状态。
- 目标 IP 和端口。
- SSH 认证方式。
- 远程端口是否已被占用。
- 代理和模型服务是否可达。

## 判断标准

超时通常优先看网络路径和防火墙。

认证失败优先看用户、密钥、密码、权限。

端口转发失败优先看远程端口占用和服务端 SSH 配置。

本地可用而服务器不可用，优先看服务器出网、代理和服务商访问限制。

## GitHub 与 deploy key 同步边界

当本地知识库需要同步到 GitHub，再让服务端 agent 读取时，要把认证问题分开看：

- SSH key 是否真的被 GitHub 接受。
- 这把 key 是个人 key、仓库 deploy key，还是另一个仓库的 deploy key。
- deploy key 是否勾选写权限。
- 本地 SSH alias 是否指向正确私钥。
- `git push` 是否使用了预期 alias。
- 服务器端是否另有自己的读权限配置。

GitHub deploy key 通常绑定单个仓库。不要因为一把 key 能访问某个仓库，就假设它能写入另一个仓库。

如果出现“GitHub 接受公钥但 push 被拒绝”，优先检查 key 类型、仓库绑定和写权限，而不是继续改 remote URL。

## 下次遇到类似问题时提醒我

不要把“我本地能用”直接等价为“服务器也能用”。先画清楚流量从哪里出去。

## 第一性原理校验

这张卡能指导下一次相似项目：当问题发生在本地、VPN、远程服务器、代理、服务和接口之间时，用分层定位避免无效试错。

它不适用于纯本地、无网络、无部署链路的任务。
