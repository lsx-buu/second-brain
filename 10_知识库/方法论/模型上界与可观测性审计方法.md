# 模型上界与可观测性审计方法

## 这张卡解决什么问题

预测模型效果不好时，人很容易继续加结构、换 backbone、引入 oracle、接入外部大模型或堆专家模块。

但 TS-CLIP 暴露出的上位问题是：错误理论上能修，不代表当前可见信息能学；弱先验有方向，不代表它能直接做预测。

原来的 oracle 上界、可观测性缺口、时序失败机制、弱先验降级，本质上是一套模型可靠性审计方法。

## 总原则

先审计错误机制，再决定是否训练复杂模块。

不要把理论空间误当成可部署能力。

不要把弱信号误当成主预测器。

不要把局部改善误当成正式指标改善。

## 五层审计

### 固定评价口径

先确认正式指标是什么。

局部点改善、row-average 改善、少数样本改善，都不一定代表正式 macro 指标改善。

评价口径不固定，所有实验都会变成讲故事。

### 定位困难区

不要平均看所有样本。

要找峰值、长 horizon、高波动、节假日、边界时段、core-hard、slope-hard 等困难区。

真正的问题通常藏在 hard regime 里。

### 做 oracle 上界

oracle 用来问：如果未来答案以某种方式被知道，最多能改善多少？

如果上界很小，这个错误类型不值得训练复杂模块。

如果上界很大，也只能说明理论空间存在，不能说明当前方法可部署。

### 做可学习性审计

可学习性回答的是：在真实决策时刻，当前可见信息能不能稳定学到修正方向、幅度或场景。

oracle 强但可学习性弱时，应该停在诊断结论，不要上线。

### 判断可观测性缺口

如果 dirty oracle 很强，clean 方法都接近同一上限，错误又集中在外部事件、节假日、运营调整、活动、天气等状态，就要怀疑可观测性缺口。

这时主线不应是“再造一个模型”，而应是“当前输入缺少解释未来 hard case 的外部动态信息”。

## 弱先验的正确位置

Chronos、Sundial、TSFM、专家经验、历史相似样本这类弱先验，不一定要直接进入主预测。

更稳的用法是把它们降级为：

- 方向信号。
- 不确定性信号。
- 风险提示。
- 小幅残差修正提案。
- 何时不要相信主模型的 guard。

弱先验要进入主系统，必须证明峰值不变差、长 horizon 不变差、多切分稳定，并且不只是复杂度换来的偶然收益。

## 失败信号

- oracle 很强，但 clean 特征学不到。
- validation 好看，test 不稳。
- row-level 指标改善，正式 macro 变差。
- 模型结构更复杂，但只改善少数切分。
- hard regime 被 overall 平均掩盖。
- 弱先验被直接当作强预测分支。
- 外部变量缺失，却继续在内部特征里打转。

## 可迁移判断

模型失败不是一个单点事实，而是一组机制问题：

- 错误在哪些样本出现。
- 理论上能不能修。
- 当前信息能不能学。
- 正式指标会不会受伤。
- 是否缺少外部可观测变量。
- 弱先验应该做预测器，还是只做可靠性信号。

## 适用场景

- 时序预测。
- 客流预测。
- 策略模型。
- 推荐系统。
- 业务预测。
- 多模态模型后处理。
- 任何存在 oracle、弱先验、hard case 和评价口径冲突的实验项目。
