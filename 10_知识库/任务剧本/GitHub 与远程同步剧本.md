# GitHub 与远程同步剧本

## 什么时候读

用户要把本地知识库、项目或配置同步到 GitHub，并让服务器或远程 agent 读取。

尤其适用于 SSH、deploy key、远端 push、服务器 clone 或权限失败。

- [[10_知识库/方法论/方法群/工程运行方法群|工程运行方法群]]

## 第一轮动作

1. 先确认本地仓库状态和 remote。
2. 判断使用的是个人 SSH key、deploy key 还是 SSH alias。
3. 测试 GitHub 是否接受当前 key。
4. 如果能读不能写，检查 deploy key 是否绑定当前仓库并允许写入。
5. 本地同步成功后，再考虑服务器如何 clone 或 pull。
6. 服务端 agent 读取时，先读仓库根目录的 `AGENTS.md` 和 Agent 启动上下文。

## 不要做什么

- 不要假设一把 deploy key 能跨仓库复用。
- 不要把 remote URL 当成唯一问题。
- 不要把本地能访问等价为服务器也能访问。
- 不要提交 Obsidian 视图状态、缓存或临时文件。
- 不要把密钥、token、账号凭证写进知识库。

## 有效信号

- 本地 `main` 跟踪远端 `main`。
- 远端分支提交号可验证。
- 服务端 clone 入口清楚。
- agent 知道从 `AGENTS.md` 和启动上下文开始读。

## 失败信号

- GitHub 接受公钥但 push 仍失败。
- key 实际属于另一个仓库。
- deploy key 没有写权限。
- 服务端看不到本地知识库。
- 知识库里混入凭证或本地视图状态。

## 写回判断

如果这次排障暴露了新的认证层边界，补到 [[10_知识库/方法论/网络与远程环境排障方法|网络与远程环境排障方法]]。

具体 key 文件名和一次性命令不进入长期知识层。
