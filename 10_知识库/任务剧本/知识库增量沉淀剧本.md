# 知识库增量沉淀剧本

## 什么时候读

用户要求把一段时间内的 Codex 对话、文档、聊天记录或项目过程沉淀到知识库。

尤其适用于用户说：

- “沉淀到知识库。”
- “提取高维方法论。”
- “根据我的反馈判断是否有效。”
- “不要存流水账。”

## 第一轮动作

1. 定位上次沉淀时间或上次提交。
2. 收集增量材料。
3. 按主题聚类，不按文件或会话逐条归档。
4. 抽取用户正向和负向反馈。
5. 判断哪些反馈能变成下次 agent 的行动约束。
6. 优先补旧方法卡，其次补索引，最后才新建卡。

## 不要做什么

- 不要保存完整聊天记录。
- 不要把项目路径、文件名和一次性脚本沉淀成长期知识。
- 不要因为内容多就新建一堆同级节点。
- 不要把用户明确否定过的输出形式再次固化。

## 有效信号

- 新卡离开原项目仍然有指导意义。
- 旧卡被补强，而不是被重复新建。
- 用户反馈被转成了判断动作。
- 索引能把新方法路由出来。

## 失败信号

- 新增很多项目现场节点。
- 方法卡读起来像总结稿，没有下一步动作。
- agent 下次仍然不知道先读哪张。
- 一张卡删掉原始文件后就失去意义。

## 写回判断

任务结束时检查：

- 是否新增了可复用方法。
- 是否命中验证队列。
- 是否需要更新 [[90_索引/Agent 启动上下文|Agent 启动上下文]]。
- 是否需要更新 [[10_知识库/偏好/AI 协作偏好总览|AI 协作偏好总览]]。
- 是否需要同步 GitHub，方便服务端 agent 读取。
