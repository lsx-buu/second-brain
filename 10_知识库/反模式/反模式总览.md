# 反模式总览

## 这张卡解决什么问题

反模式不是为了记录 agent 做错了什么。

它的作用是让下一个 agent 更快识别：当前任务已经偏离用户真正目标，应该停止惯性执行，切换到正确方法。

## 反模式一：没有原文就总结

表现：

- 只有标题、作者简介、页面片段，就开始写知识卡。
- 人物目录看起来完整，但没有真实技巧、对白或原文。
- 用户反馈“太浅”“没有干货”“没搜到真正心得就填库”。

应该切换到：

- [[10_知识库/任务剧本/内容型知识库采集剧本|内容型知识库采集剧本]]
- [[10_知识库/方法论/原文优先的内容型知识库采集方法|原文优先的内容型知识库采集方法]]

## 反模式二：把生成图当业务证据

表现：

- 画面好看，但无法证明真实业务场景。
- 真实素材和功能点不相关。
- 没有现场素材时，仍然硬塞生成画面。
- 用户反馈“配图和功能点没有关联”。

应该切换到：

- [[10_知识库/任务剧本/汇报图与真实素材融合剧本|汇报图与真实素材融合剧本]]
- [[10_知识库/方法论/生成视觉与真实证据的分工方法|生成视觉与真实证据的分工方法]]

## 反模式三：自动化把监督成本转嫁给用户

表现：

- 脚本一直点错按钮、发错输入框、跳错页面。
- 用户反复提醒“你没点开评论”“你又点到收藏了”。
- 用户说“我让你和脚本协同，不是让我和脚本协同”。

应该切换到：

- agent 先判断页面状态，再让脚本执行重复动作。
- 脚本只负责稳定动作，不负责替代所有判断。
- 如果流程错三次，停止盲跑，重写流程假设。

关联方法：

- [[10_知识库/方法论/情绪反馈驱动的 Agent 有效性校准|情绪反馈驱动的 Agent 有效性校准]]
- [[10_知识库/任务剧本/内容型知识库采集剧本|内容型知识库采集剧本]]

## 反模式四：项目节点离开现场文件就失效

表现：

- 卡片主要记录路径、脚本名、文件状态。
- 文件一删，知识节点就没有意义。
- 项目层堆了很多兄弟节点，但没有更高维判断。

应该切换到：

- [[10_知识库/任务剧本/知识库增量沉淀剧本|知识库增量沉淀剧本]]
- [[10_知识库/方法论/从项目到方法论的提炼协议|从项目到方法论的提炼协议]]
- [[10_知识库/方法论/从具体项目提炼高维能力|从具体项目提炼高维能力]]

## 反模式五：课程设计过度开放

表现：

- 给孩子很多项目选择，但老师无法统一教学。
- 课程看起来有趣，但课堂不可控。
- 用户反馈“可以选题材，但是开始做不能太自由”。

应该切换到：

- [[10_知识库/任务剧本/课程与训练营设计剧本|课程与训练营设计剧本]]
- [[10_知识库/方法论/可控创造力的课程设计方法|可控创造力的课程设计方法]]

## 反模式六：先润色论文，再决定投哪里

表现：

- 花时间改稿、改格式，但目标期刊约束不清楚。
- 不知道分区、审稿周期、模板、数据开放要求。
- 导师沟通时说不清为什么投这个。

应该切换到：

- [[10_知识库/任务剧本/论文投稿准备剧本|论文投稿准备剧本]]
- [[10_知识库/方法论/论文投稿目标约束方法|论文投稿目标约束方法]]

## 反模式七：连接问题混成一团

表现：

- remote、SSH key、deploy key、服务器 clone、GitHub 权限混在一起猜。
- 本地能用，就默认服务器也能用。
- 一把 key 能读某仓库，就假设能写另一个仓库。

应该切换到：

- [[10_知识库/任务剧本/GitHub 与远程同步剧本|GitHub 与远程同步剧本]]
- [[10_知识库/方法论/网络与远程环境排障方法|网络与远程环境排障方法]]

## 使用规则

如果用户连续两次指出同一类问题，不要继续微调产物。

先判断命中了哪个反模式，再切换剧本。

如果反模式反复出现，就应该更新 [[90_索引/Agent 启动上下文|Agent 启动上下文]] 或 [[10_知识库/偏好/AI 协作偏好总览|AI 协作偏好总览]]。
